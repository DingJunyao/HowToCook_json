# parse-ingredients 技能

## 目标
给定一个食材列表，合并相同的食材为标准化格式的 JSON 数据，用于导入数据库。

## 输入
- **第一个参数**：要导出 JSON 文件的完整路径（必须严格使用此路径）
  - 例如：`/home/ding/code/HowToCook_json/out/ingredients.json`
  - 例如：`out/ingredients.json`
- 第二个参数（通过 stdin）：一个食材列表（可能来自多个菜谱解析结果，包含食材的名称、单位）

## 输出
- **必须**在第一个参数指定的完整路径导出标准化的食材 JSON 数组，格式如下：
- **不要**使用其他默认名称如 `ingredients_standard.json` 或 `ingredients_parsed.json`
```json
[
  {
    "ingredient_name": "食材名称",
    "aliases": ["别名1", "别名2", "别名3"],
    "category": "分类",
    "unit": "默认单位"
  },
  ...
]
```

## 分类体系
- grains: 谷物
- vegetables: 蔬菜
- fruits: 水果
- meat: 肉类
- seafood: 海鲜
- eggs: 蛋类
- dairy: 乳制品
- seasoning: 调味品
- oil: 油脂
- nuts: 坚果
- beverages: 饮品
- others: 其他

## 处理规则
- 不同菜谱里面对同种食材的叫法可能不一样，需要智能识别并合并
- 为每种食材建立别名列表，包含在不同菜谱中出现的不同叫法
- 根据食材特性正确分类
- 设置合理的默认单位（如：g、mL、个、条、根等）
- 智能识别同一种食材的不同表达方式（例如：土豆/马铃薯/洋芋 为同一食材）
- 默认单位，选择使用最多的那个单位。

### ⚠️ 合并规则（禁止合并的情况）

**仅当不同名称指的是完全相同的食材时才合并**，以下情况**禁止合并**：

1. **不同部位不合并**
   - ❌ 猪肉 ≠ 五花肉 ≠ 里脊肉 ≠ 前腿肉
   - ❌ 鸡肉 ≠ 鸡胸肉 ≠ 鸡腿 ≠ 鸡翅 ≠ 鸡爪
   - 这些应分别作为独立的食材条目

2. **不同品种/种类不合并**
   - ❌ 植物油 ≠ 橄榄油 ≠ 花生油 ≠ 玉米油 ≠ 菜籽油
   - ❌ 挂面 ≠ 意大利面 ≠ 方便面 ≠ 河粉 ≠ 鲜面条
   - ❌ 芝麻 ≠ 白芝麻 ≠ 黑芝麻

3. **不同加工状态不合并**
   - ❌ 干 ≠ 湿/鲜（如：干面条 ≠ 鲜面条）
   - ❌ 熟 ≠ 生（如：熟白芝麻 ≠ 白芝麻）
   - ❌ 切片 ≠ 条 ≠ 块 ≠ 粒

4. **不同属性不合并**
   - ❌ 去皮 ≠ 不去皮
   - ❌ 去头/去骨 ≠ 完整

### ✅ 可合并的情况（仅限以下情况）

**仅限同义词、方言叫法、简写/全称的差异：**

- ✅ 土豆 / 马铃薯 / 洋芋（同一种蔬菜）
- ✅ 菠菜 / 菠菜叶（常用表达）
- ✅ 食用油 / 油（常用简称）
- ✅ 白砂糖 / 白糖（常用简称），但是绵白糖不应合并
- ✅ 花雕酒 / 黄酒 / 料酒（同一种料酒的叫法）
- ✅ 木耳 / 黑木耳（同一种菌类），但是白木耳不应合并

## 注意事项
- 由于食材叫法、归属多种多样，故请**直接**分析合并，**禁止**使用脚本合并。
- 仅处理真正的食材，排除厨具、餐具、厨房用品
- 调料（盐、酱油、醋等）也属于食材范畴
- 合并时保留所有相关的别名
- 分类尽可能准确，不确定时归类为 "others"
- **重要**：aliases 数组中**不应包含 ingredient_name 本身**（避免重复）
- **重要**：输出文件路径必须使用第一个参数传入的完整路径，不要使用其他默认名称如 `ingredients_standard.json`
